## 시스템 개요

### 데이터 구조
```
lessonMaterialTemplates (컬렉션) - 관리자가 만든 템플릿
├── templateId
    ├── title: string
    ├── code: string (J28, S28, F28 등)
    ├── sections: Array
    │   ├── id: string
    │   ├── title: string
    │   ├── order: number
    │   └── links: Array<{label, url}>
    ├── links: Array<{label, url}>
    ├── createdAt: Timestamp
    └── updatedAt: Timestamp

lessonMaterials (컬렉션) - 사용자별 수업 자료
├── materialId
    ├── userId: string
    ├── title: string
    ├── order: number
    ├── templateId?: string (템플릿 연결)
    ├── userCode?: string (사용자가 직접 추가한 경우)
    ├── createdAt: Timestamp
    └── updatedAt: Timestamp
    └── sections (서브컬렉션)
        ├── sectionId
            ├── title: string
            ├── order: number
            ├── viewUrl: string
            ├── originalUrl: string
            ├── templateSectionId?: string
            ├── createdAt: Timestamp
            └── updatedAt: Timestamp
```

### 주요 로직
1. **템플릿 자동 추가**: 사용자가 접근 가능한 템플릿을 자동으로 `lessonMaterials`에 복사
2. **템플릿 동기화**: 템플릿 제목/구조 변경 시 사용자 데이터에 반영
3. **중복 제거**: 같은 templateId를 가진 항목 중 첫 번째만 표시
4. **권한 필터링**: 활성화된 캠프 코드에 해당하는 자료만 표시

## 시나리오별 상세 분석

### 시나리오 1: 관리자가 템플릿 생성 → 사용자 자동 추가

**절차**:
1. 관리자가 "J28 반OT" 템플릿 생성 (소제목: 반OT 1, 반OT 2)
2. J28 멘토 A 접속 → 자동으로 "J28 반OT" 추가됨
3. 멘토 A가 "반OT 1"에 URL 입력
4. 관리자가 템플릿에 "반OT 3" 소제목 추가
5. 멘토 A 새로고침 → "반OT 3" 자동 추가됨

**엣지케이스**:
- ⚠️ **문제 1**: 멘토 A가 URL 입력 중에 관리자가 소제목 삭제하면?
  - **현재**: 멘토 A는 계속 보이지만 템플릿 연결 끊김
  - **영향**: 데이터 불일치

- ⚠️ **문제 2**: 멘토 A가 "반OT 1" URL 입력 중에 관리자가 "반OT 1" 제목 변경하면?
  - **현재**: 제목이 템플릿 제목으로 강제 변경됨
  - **영향**: 사용자 혼란 (내가 입력 중인데 제목이 바뀜)

### 시나리오 2: 사용자가 URL 입력 후 관리자가 템플릿 삭제

**절차**:
1. J28 멘토 A가 "반OT" 템플릿의 소제목에 URL 입력
2. 관리자가 "반OT" 템플릿 삭제
3. 멘토 A 새로고침 → "반OT" 자료가 화면에서 사라짐

**엣지케이스**:
- ⚠️ **문제 3**: 사용자 데이터가 화면에서 사라짐
  - **현재**: Firestore에는 존재하지만 접근 불가
  - **영향**: 사용자는 데이터 손실로 인식
  - **심각도**: 🔴 높음

**복구 방법**:
```javascript
// 관리자 도구로 복구 가능
// 1. lessonMaterials에서 templateId 제거
await updateDoc(doc(db, 'lessonMaterials', materialId), {
  templateId: null,
  userCode: 'J28' // 수동으로 코드 설정
});
```

### 시나리오 3: 동일 제목 템플릿 중복 생성

**절차**:
1. 관리자가 "J28 반OT" 템플릿 생성
2. 멘토 A에게 자동 추가됨
3. 관리자가 실수로 또 "J28 반OT" 템플릿 생성 (같은 제목)
4. 멘토 A에게 중복으로 추가됨

**엣지케이스**:
- ⚠️ **문제 4**: 동일 제목 자료 2개
  - **현재**: 중복 제거 로직이 templateId 기반이므로 둘 다 보임
  - **영향**: 사용자 혼란
  - **심각도**: 🟡 중간

### 시나리오 4: 네트워크 오류 중 저장

**절차**:
1. 사용자가 URL 입력 후 저장 버튼 클릭

3. 사용자가 모르고 페이지 이동

**엣지케이스**:
- ⚠️ **문제 5**: 저장 실패 시 알림 없음
  - **현재**: try-catch로 에러 처리하지만 사용자 알림 불확실
  - **영향**: 데이터 손실
  - **개선안**: toast 알림 또는 저장 완료 메시지 명확히

### 시나리오 5: 템플릿 링크와 사용자 URL 동시 존재

**절차**:
1. 관리자가 템플릿 소제목에 링크 추가 (예: "구글 드라이브 폴더")
2. 사용자가 해당 소제목에 viewUrl/originalUrl 입력
3. 화면에는 어떻게 표시되나?

**엣지케이스**:
- ⚠️ **문제 6**: 표시 우선순위 불명확
  - **현재**: 템플릿 links는 별도 버튼으로 표시, URL은 입력 필드
  - **영향**: 사용자가 둘의 차이를 이해하기 어려움
  - **개선안**: UI에서 명확히 구분 ("관리자 참고 링크" vs "내 업로드")

## 발견된 문제 및 개선 사항

### 🔴 심각한 문제 (즉시 수정 필요)

1. **템플릿 삭제 시 사용자 데이터 손실처럼 보임**
   - **원인**: templateId 연결된 자료는 필터링에서 제외
   - **해결**: 
     - 방법 1: Soft delete (템플릿에 `deleted: true` 플래그)
     - 방법 2: 삭제 전 "N명 사용 중" 경고
     - 방법 3: templateId 없어도 userCode로 표시되도록 변경

2. **템플릿 코드 변경 시 사용자 데이터 접근 불가**
   - **원인**: 코드 기반 필터링
   - **해결**: 코드 변경 금지 또는 변경 시 강제 경고

3. **Google Drive 비공개 링크 접근 불가**
   - **원인**: URL validation 없음
   - **해결**: 공유 권한 체크 또는 안내 메시지

### 🟡 중간 문제 (개선 권장)

4. **동일 제목 템플릿 중복 생성 가능**
   - **해결**: 제목 중복 체크 또는 "(복사)" 자동 추가

5. **코드 미선택 템플릿 생성 가능**
   - **해결**: 코드 필수 입력 validation

6. **링크 입력 불완전 시 자동 제거**
   - **해결**: 저장 전 validation 알림

7. **템플릿 소제목 삭제 시 사용자 데이터 불일치**
   - **해결**: 삭제된 소제목은 사용자 화면에서도 숨김 처리

### 🟢 경미한 문제 (UX 개선)

8. **활성 캠프 없을 때 안내 부족**
   - **해결**: "활성화된 캠프가 없습니다" 메시지

9. **URL 형식 검증 없음**
   - **해결**: URL validation 추가

10. **저장 완료/실패 알림 불명확**
    - **해결**: toast 메시지 명확히
